# Docker и деплой

Дата: 2026-06-17
Статус: реализован

## Цель

Упаковать приложение в один Docker-образ, который стартует автоматически и слушает
порт из переменной окружения `PORT`. После деплоя — публичная ссылка.

## Ключевое решение: один образ, один PORT

Приложение состоит из SPA-фронта и FastAPI-бэка. Требование «один образ по `PORT`»
решаем так: **бэкенд отдаёт и API, и собранный фронт** с того же origin.

- Фронт собирается с `VITE_API_BASE_URL=""` → обращается к API по относительным путям
  (same-origin), без CORS и без отдельного адреса.
- FastAPI монтирует `/assets` (StaticFiles) и отдаёт `index.html` на все клиентские
  маршруты (`/`, `/admin`, `/book/...`). API-роуты зарегистрированы раньше catch-all,
  поэтому имеют приоритет.
- Статика подключается только если каталог `app/static` существует (в dev его нет —
  поведение бэкенда не меняется, тесты и mock-разработка работают как прежде).

## Dockerfile (multi-stage)

1. `node:20-alpine` — `npm ci` + `npm run build` фронта → `frontend/dist`.
2. `python:3.12-slim` — установка `backend/requirements.txt`, копирование `app/` и
   `frontend/dist` → `app/static`.
3. Старт: `uvicorn app.main:app --host 0.0.0.0 --port ${PORT:-8000}` (shell-форма, чтобы
   подставился `PORT`). Контейнер стартует приложение автоматически.

## Деплой

- `render.yaml` (Blueprint): web-сервис, runtime docker, healthcheck `/owner`. Render
  сам прокидывает `PORT`.
- Запасной вариант — Railway (тот же Dockerfile, автодетект, публичный домен).
- Инструкции и место для публичной ссылки — `DEPLOY.md`.

## Проверка

- `docker build` — образ собирается.
- `docker run -e PORT=3000 -p 3000:3000` — приложение стартует на заданном порту;
  `GET /owner`, `GET /event-types` отвечают; `/` отдаёт SPA; `/admin` → index.html (200).
- Браузер против контейнера: UI грузит реальные данные из same-origin API.

## Результат

`Dockerfile` в корне; приложение запускается в контейнере по `PORT`; готовы конфиг и
инструкция для публикации на Render/Railway.
